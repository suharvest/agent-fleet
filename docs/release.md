# AgentFleet Release And Distribution

AgentFleet is distributed as prebuilt release archives plus a small
install script. Normal users should not need Rust installed locally.

## User Install From Release

Each `v*` tag publishes platform archives:

```text
agent-fleet-linux-x86_64.tar.gz
agent-fleet-linux-aarch64.tar.gz
agent-fleet-macos-x86_64.tar.gz
agent-fleet-macos-aarch64.tar.gz
agent-fleet-windows-x86_64.zip
```

macOS/Linux:

```bash
tar -xzf agent-fleet-<platform>.tar.gz
cd agent-fleet-<platform>
./scripts/install.sh
```

Windows PowerShell:

```powershell
Expand-Archive .\agent-fleet-windows-x86_64.zip
cd .\agent-fleet-windows-x86_64\agent-fleet-windows-x86_64
.\scripts\install.ps1
```

The install script uses the bundled `fleet`/`fleet.exe` binary and runs:

```bash
fleet doctor --fix --write-shell-profile
```

It installs shims into `~/.rpty/bin`, copies the bundled Fleet backend example
files, discovers local Agent commands, and updates the shell profile.

## Local Install From Source

macOS/Linux:

```bash
git clone <repo-url>
cd agent-fleet
./scripts/install.sh
```

Windows PowerShell:

```powershell
git clone <repo-url>
cd agent-fleet
.\scripts\install.ps1
```

When no prebuilt binary is present, the script builds `target/release/fleet`,
installs the runtime into `~/.rpty/bin/fleet-router`, creates `fleet`, `rpty`,
and `bash` shims, discovers known Agent commands, and writes the PATH entry to
the user's shell profile. On Windows, shims are installed as `.cmd` files and
the PowerShell profile is updated.

## Tag Release CI

Release artifacts are generated by `.github/workflows/release.yml`.

Create and push a version tag:

```bash
git tag v0.1.0
git push origin v0.1.0
```

The workflow runs tests, builds release binaries, packages the runtime files,
uploads checksums, and publishes a GitHub Release.

Each archive contains:

```text
fleet and rpty binaries
scripts/install.sh
scripts/install.ps1
README.md
docs/getting-started.md
docs/release.md
fleet_backend/bootstrap.sh
fleet_backend/devices.example.json
fleet_backend/fleet.py
fleet_backend/pyproject.toml
fleet_backend/uv.lock
```

## Fleet Backend And Device Inventory

The Rust-native Fleet backend is built into the binary. Legacy compatibility
files are bundled in:

```text
fleet_backend/fleet.py
```

The private device inventory is not committed. Users should create:

```text
~/.rpty/bin/fleet_backend/devices.json
```

from:

```text
~/.rpty/bin/fleet_backend/devices.example.json
```

Release artifacts should include the Rust binary, `bootstrap.sh`, and the
example inventory. Real device hosts, usernames, ports, tags, and passwords stay
in the private `devices.json` only.

Python is not required for normal Fleet use. For compatibility testing,
`RPTY_FLEET_NATIVE=0` disables native Fleet commands and forces the legacy
fallback backend.

Users with an existing external Fleet backend can still configure:

```bash
fleet config --fleet-py /path/to/fleet.py
```

or:

```bash
fleet config --fleet-hub /path/to/hub
```

The config file is:

```text
~/.rpty/config.toml
```

Environment variables still override config:

```bash
RPTY_FLEET_PY=/path/to/fleet.py
RPTY_FLEET_HUB=/path/to/hub
```

## Agent Discovery

Built-in Agent names:

```text
codex
claude
opencode
```

Add another Agent command:

```bash
fleet config --agent gemini
fleet doctor --fix
```

`doctor --fix` installs a same-name shim when the real Agent command is present
later in `PATH`.

## Upgrade

From source:

```bash
git pull
./scripts/install.sh
```

From a downloaded binary:

```bash
./fleet doctor --fix --write-shell-profile
```
